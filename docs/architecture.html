BabyBedtimeTeller — Architecture & Data Flow

How the app is put together, why it's built this way, and where the risk actually lives. The short version: there is no backend — the entire product is one static HTML file, one external API call, and a handful of client-only fallbacks that keep working when that call fails.

Design tradeoffs

No backend, ever

Everything — form logic, the API call, narration, downloads, fallback content — runs in one static HTML file. That makes the app free to host and trivial to deploy, but trades away anything that needs a server: hiding a secret, metering usage, or syncing data across devices. All three were out of scope for a personal tool, so the trade was easy.

Bring-your-own API key, stored client-side

Rejected hardcoding a key into the repo (leaks in public source) and rejected a secret-holding proxy (reintroduces a server for a hobby project). Landed on: paste your key in, it lives only in that browser's localStorage. Visible in that browser's dev tools while in use — fine for one person, wrong for a multi-tenant product.

Sonnet, thinking disabled

A 350-word bedtime story doesn't need an extended-reasoning pass. Skipping it and using Sonnet over Opus trades a sliver of prose polish for meaningfully lower latency and cost on every request — the right call against a "under 10 seconds, kid is waiting" target.

No streaming

max_tokens: 700 is well under the point where streaming is needed to dodge timeouts. Skipping it simplifies the state machine at the cost of a slightly longer pause before anything renders — acceptable since narration can't start until the story is complete anyway.

Soft guardrails, not a hard filter

Content safety leans on the system prompt and Claude's own judgment (a "scary monster" becomes gentle) rather than a client-side keyword blocklist. Less deterministic, far less brittle — and a blocklist would always be incomplete anyway.

Technical insights

The trust boundary is the whole story

Normally the client/server line is where secrets live. With no server, the only boundary that matters is "this browser" vs. "the internet" — and the one thing crossing it, the API key over HTTPS straight to Anthropic, is exactly the one thing that must never sit in committed source.

The browser-access header is sanctioned, not a hack

anthropic-dangerous-direct-browser-access is Anthropic's own opt-in for calling the API straight from a browser. It's the supported path for a single-user tool like this, not an unsupported workaround.

Client-only fallbacks compound reliability for free

TTS, downloads, and the 5-story fallback bank all work with the network off. Since avoiding a backend already forces everything client-side, the "still basically works" path costs nothing extra to build — the app ends up more resilient than its one external dependency suggests.

DOM injection is structurally prevented

Story text and the child's name are inserted via textContent, never innerHTML — so even an adversarial theme string can't execute as markup, regardless of how thorough the input sanitization happens to be. Defense in depth, not a single filter.

The error/fallback path was verified, not assumed

Before shipping, the full retry → fallback flow was driven end-to-end with a fake API key via a scripted browser. That caught a real bug — a CSS rule silently overriding the hidden attribute on the "key saved" banner — that a code read-through alone hadn't surfaced.
